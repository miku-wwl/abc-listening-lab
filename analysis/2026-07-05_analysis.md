## 3. 新问题 vs 旧问题

### 重复出现的问题

- 滚动 VTT 去重、中间长空白和结尾零散无上下文字幕。
- 技术缩写和产品名音形不稳定：6LoWPAN、BLE、RF、VNC、CoAP、MQTT、ADC、GPIO、LoRa。
- 数字对象配对：channel 5、11、16、25、26；5 秒与 30 秒；10–100 米；20 公里及理想条件下更远距离；一次每小时与十年电池寿命。

- 组件和数据流关系：SensorTag、LaunchPad、serial、Raspberry Pi、Node-RED、dashboard、VNC、cloud。

### 新出现的问题

- 排障对话中的证据强度：`I think / maybe / assume / my guess` 是假设，不是结论；只有改 channel 后节点出现才形成较强证据。
- 无线标准范围与配置值的区别：配置双方一致仍可能使用非法或不支持的 channel。
- 配置周期与端到端可见延迟不是同一个数字；pipeline、服务器和页面渲染都会增加延迟。
- “本地服务能运行”与“海外队友能安全访问”是两个不同问题。
- 作业设计需要从“能显示数据”升级到“对场景有用”：历史分析、预测、阈值通知和明确用户。

## 7. 复盘问题

1. 学生最初把 channel 配成多少，指导值是多少？
   答：学生用了 channel 5，指导值是 channel 16。
2. 为什么“SensorTag 和 LaunchPad 使用相同 channel”仍不足以保证工作？
   答：双方一致只是一个条件；所选 channel 还必须属于所用 IEEE/6LoWPAN 配置支持的范围。对话中归纳的范围是 11–26。
3. 本次数据路径是什么？
   答：SensorTag 经 6LoWPAN 到 LaunchPad，再经 serial 到 Raspberry Pi，随后进入服务或 Node-RED flow，最后到 dashboard。
4. BLE 和 6LoWPAN 在对话中是什么关系？
   答：它们被明确区分为不同的低功耗无线网络技术；本实验主要使用 6LoWPAN。
5. 为什么必须使用课程提供的 Raspberry Pi image？
   答：默认 Raspbian image 缺少课程流程依赖的预装文件和配置。
6. 对话中怎样从“观察”走到“根因”？
   答：先观察 sensor table 没有 mote；重做并修改 channel 后节点和 packets 出现；再核对 IEEE channel 范围，因此把 channel 配置确定为根因。
7. 为什么把更新间隔改成 5 秒后，页面不一定严格每 5 秒变化？
   答：数据还要经过采集、无线传输、服务器、pipeline 和页面渲染，端到端延迟不等于单个配置值。
8. 教师为什么建议先确定 use case，而不是立刻买更多 sensors？
   答：现有数据已经足够做 dashboard、analytics、prediction 或 alerts；新增传感器可能引入 GPIO、BLE、驱动和新数据管线，增加范围并让部分现有工作失去价值。
9. `6LoWPAN`、`LoRa`、`Node-RED` 最容易被 ASR 分别写成什么？
   答：类似 `six load fan / low pen`、`Laura`、`no red`。
10. 这份材料最适合怎样切分训练？
    答：先练 00:20–00:23 的协议关系、00:56–01:06 的根因排查、01:22–01:37 的 Node-RED 配置、01:39–01:48 的低功耗和 LoRa 场景；每段控制在 8–12 分钟。
