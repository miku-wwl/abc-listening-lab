## 3. 新问题 vs 旧问题

### 重复出现的问题

1. **技术缩写和硬件名被熟词吸附**：`6LoWPAN` 被写成 `low pan`，`serial cable` 可能被写成 `serial table`，`SensorTag / CC2650 / Raspberry Pi` 也识别不稳。这个和 2026-07-05 的 `6LoWPAN / Node-RED / LoRa / CoAP` 是同一类问题。
2. **自然口语重启造成“我是不是漏了”的错觉**：`what you what the the uh`、`how it's going to help and what's going to help` 不是你的耳朵坏了，是课堂口语本来就这样，文明在这里确实有点倒退。
3. **事实、建议、假设混在一起**：`I assume APA`、`I have to speak to Negin`、`don't bother doing anything yet` 都需要区分“已确认规则”和“暂时建议”。这延续了 2026-07-05 的排障 fact / hypothesis 问题。
4. **条件句决定行动**：`if you don't have a serial cable`、`if you've got a monitor`、`once you have the IP address` 都是实际 setup 决策的开关。
5. **作业要求中的抽象结构词容易被忽略**：`project scope / business case / value / visualize data / what-how-when-who` 不像技术名那么显眼，但决定 assignment 怎么写。

### 新出现的问题

1. **作业结构听力**：这次首次集中出现 `project scope / part one / business case / what-how-when-who / referencing / APA`。这是 academic admin + technical project 的混合语域。
2. **暂缓操作类指令**：`don't bother doing anything with the Raspberry Pi yet` 这种话很重要，因为它不是教你做，而是告诉你现在别做。人类课程的高阶智慧之一：有时最正确的操作是停止折腾。
3. **headless setup 替代路径**：同一目标有两条路径，serial cable/headless 或 monitor/Wi-Fi/SSH。听力上要恢复 decision tree，而不是只记一串设备名。
4. **business value 的要求**：不是随便采集数据，而是解释数据如何被使用、商业或社区价值在哪里。

### 已经改善的问题

目前材料不足，暂时无法判断真实改善情况。原因很朴素：没有裸听 notes，就不知道你到底卡在哪里；仓库能看到“材料风险”，看不到“你的真实错误”。不过从训练材料选择看，7 月 7 日比 7 月 5 日的 114 分钟巨兽更短、更聚焦，这是进步。耳朵终于不用被当成云服务器压测了。
